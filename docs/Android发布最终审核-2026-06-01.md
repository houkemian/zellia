# Android 发布最终审核 - 2026-06-01

## 审核结论

当前版本暂不建议直接发布 Android 正式版。

本次审核前提：iOS 暂不发布，iOS 相关配置、构建、Firebase、RevenueCat、App Group、小组件能力均不作为本次发布阻断项。

Android 侧已有基础功能、生产 API 默认地址、Firebase Android 配置和测试用例通过。正式签名配置在打包电脑上存在，本仓库不提交密钥文件；发布前仍需在打包电脑确认 release 签名生效。当前主要剩余项是静态检查、后端部署验证和 Android release 构建回归。

## 发布阻断项

### 1. Android release 正式签名需在打包电脑验证

位置：

- `mobile/android/app/build.gradle.kts`
- `mobile/android/key.properties`

现状：

- `build.gradle.kts` 会读取 `mobile/android/key.properties`。
- 本审核环境未发现 `mobile/android/key.properties`，但该文件属于本地密钥配置，不应提交到仓库。
- 用户确认打包电脑上已存在 `key.properties`。
- 如果打包电脑上的 `key.properties` 有效，`release` 构建会使用正式 signingConfig；如果缺失或为空，当前 Gradle 配置会退回 debug 签名。

风险：

- 风险不在仓库缺少 `key.properties`，而在最终打包环境是否确实使用正式签名。
- 如果打包电脑配置错误，生成的 APK/AAB 可能仍是 debug 签名，不能用于 Google Play 正式发布。
- 后续如果更换 keystore，会影响升级链路。

发布前要求：

- 在打包电脑确认 `mobile/android/key.properties` 存在且包含 `storeFile`、`storePassword`、`keyAlias`、`keyPassword`。
- 确认 `storeFile` 指向的正式 keystore 文件存在且可读。
- 执行 release AAB 构建后，检查产物签名证书指纹与 Google Play / Firebase / Google Sign-In 后台登记一致。
- 保持 `key.properties` 和 keystore 不入库。


结果：

其他主要类别：

- 多处 `curly_braces_in_flow_control_structures`。

风险：

- `BuildContext` 跨 async gap 在真实设备上可能导致页面销毁后继续使用 context，引发异常或错误弹窗。
- deprecated API 不一定立即崩溃，但会增加后续 Flutter / SDK 升级成本。

发布前最低要求：

- 至少清理 warning，让 `flutter analyze` 通过。
- 建议优先处理 `use_build_context_synchronously` 出现在支付、家庭、今日首页、报表导出等关键路径的地方。

### 3. 后端 Firebase token 项目校验已修复，仍需部署验证

位置：

- `backend/app/dependencies.py`
- `backend/app/routers/auth.py`
- `backend/app/config.py`

已完成修复：

- 登录链路和业务接口鉴权链路不再使用 token 自身的 `aud` 作为 Google 公钥校验的信任根。
- Google 公钥 fallback 只使用服务端配置的 `settings.firebase_project_id`。
- 如果 Firebase token 的 `aud` 与服务端配置的 `FIREBASE_PROJECT_ID` 不一致，登录接口返回 401，业务鉴权不会接受该 token。
- `ZELLIA_ENV=production/prod` 时，后端启动阶段强制要求配置 `FIREBASE_PROJECT_ID`。

剩余发布前验证：

- 生产环境必须设置 `ZELLIA_ENV=production` 和固定 `FIREBASE_PROJECT_ID`。
- 用本项目 Firebase ID token 验证登录、注册后登录、业务接口鉴权均正常。
- 用其他 Firebase project 的 ID token 验证 `/auth/firebase-login` 返回 401。
- 在 Firebase Admin SDK 不可用、但 `FIREBASE_PROJECT_ID` 已配置的环境下，Google 公钥 fallback 仍只接受本项目 token。

### 4. 后端 `SECRET_KEY` 生产默认值风险已修复，仍需部署配置

位置：

- `backend/app/config.py`
- `backend/.env.example`

已完成修复：

- 新增 `ZELLIA_ENV` 配置，默认 `local`，`.env.example` 已补充。
- `ZELLIA_ENV=production` 或 `ZELLIA_ENV=prod` 时，如果 `SECRET_KEY` 为空、仍是默认值/示例占位值、或长度小于 32 字符，后端会在配置加载阶段拒绝启动。
- 同一生产校验还会要求 `FIREBASE_PROJECT_ID` 已配置，避免 Firebase 项目校验退化。

剩余发布前验证：

- 生产部署必须显式设置 `ZELLIA_ENV=production`。
- 生产部署必须设置长度不少于 32 字符的高强度 `SECRET_KEY`。
- 在部署流水线中增加启动验证，确认缺失、默认或示例占位 `SECRET_KEY` 时服务无法启动。
- legacy JWT 兜底仍存在，因此 `SECRET_KEY` 需要纳入密钥轮换和泄露响应预案。

### 2. Android 权限声明需要对应商店说明

位置：

- `mobile/android/app/src/main/AndroidManifest.xml`

当前权限：

- `CAMERA`：扫码绑定家人二维码。
- `INTERNET`：访问后端 API。
- `RECORD_AUDIO`：录制亲情语音服药提醒。
- `POST_NOTIFICATIONS`：服药提醒、家庭提醒、异常通知。
- `SCHEDULE_EXACT_ALARM`：尽量按时触发服药提醒。

风险：

- Google Play 对敏感权限、通知、精确闹钟会要求说明使用场景。
- `SCHEDULE_EXACT_ALARM` 需要确认目标市场与 Google Play 政策是否允许该健康提醒场景。

发布前要求：

- 商店隐私表单中准确声明相机、麦克风、通知用途。
- 精确闹钟权限需准备说明：用于用户主动创建的服药提醒。
- 应用内首次触发权限请求前，应有足够清晰的上下文说明。

### 3. 硬编码中文文案较多

现状：

- `mobile/lib` 中仍有大量中文硬编码。
- 部分代码使用 `_text(...)` 或 `_textForLocale(...)` 做中英切换，但并未完全进入 ARB 本地化体系。

风险：

- 若首发仅中文市场，影响较小。
- 若上架面向英文用户，文案一致性和可维护性不足。

建议：

- 正式面向多语言市场前，统一迁移到 `l10n` ARB。
- 本次 Android 中文首发可不作为阻断。

### 4. Debug 日志较多

现状：

- 多个服务和页面存在 `debugPrint`。
- 大部分属于 Flutter debug 输出，release 下影响较小，但仍需避免输出敏感健康数据、token、上传 URL。

建议：

- 发布前抽查登录、家庭、语音上传、支付、推送链路日志。
- 禁止输出 access token、Firebase token、R2 presigned URL、完整健康记录明细。
