# Zellia APP 产品完善度与一人公司改进报告

> 日期：2026-06-04  
> 视角：产品成熟度、功能完整性、商业化可行性、一人公司执行优先级  
> 依据：当前仓库代码与既有产品文档，包括 `mobile/lib/screens`、`mobile/lib/services`、`backend/app/routers`、`PRODUCT_OVERVIEW.md`、`PRODUCT_STRATEGY_ANALYSIS.md`、`CODE_RISK_REVIEW.md`、`PAY_WALL.md`。

## 1. 结论

Zellia 当前已经达到 **MVP+ 到早期 Beta** 的产品阶段，不再是单点功能验证。它已经具备较完整的家庭慢病管理闭环：

- 长辈每天完成用药打卡与血压/血糖记录。
- 家属通过绑定、审核、家属视角查看、提醒和周报参与协同。
- PRO 通过家庭共享、亲情语音、周报、临床摘要、PDF 报告承载商业化价值。
- Firebase Auth、legacy JWT、RevenueCat、FCM、Redis、R2、HomeWidget 等关键基础能力已有实现痕迹。

但如果从“能否作为一人公司稳定上线并持续运营”的角度看，当前还不应继续横向堆功能。更重要的是补齐 **安全、弱网、付费一致性、数据生命周期、指标和客服兜底**。这些不是锦上添花，而是健康类产品进入真实用户家庭后会直接影响信任和留存的基础能力。

总体判断：

| 维度 | 当前判断 | 产品含义 |
| --- | --- | --- |
| 核心功能完整度 | 较高 | 用药、体征、家庭协同、报告、PRO 已形成闭环 |
| 银发友好体验 | 中到较高 | 有大字号、大按钮、低层级首页思路，但功能增多后需继续减负 |
| 商业化可测试性 | 较高 | PRO 权益方向合理，付费对象应优先是子女/照护者 |
| 生产稳定性 | 中 | 弱网、超时、同步、配置、支付 webhook 等仍需加固 |
| 医疗数据可信度 | 中 | 用药计划已软删除，但体征、日志、账号删除仍需更谨慎 |
| 一人公司适配度 | 中到较高 | 适合小范围上线验证，但需要削减复杂度并建立运营兜底 |

一句话结论：

> 功能上已经足够支撑小规模真实用户试点；但要作为一人公司正式商业化，应优先补“可信、稳定、可运营”，而不是继续加新模块。

## 2. 当前已具备的产品能力

已实现或已有代码支撑：

- 邮箱注册/登录。
- Firebase 登录与后端 `/auth/firebase-login` 对接。
- legacy JWT 兜底。
- 当前用户资料读取与更新。
- 家属为长辈代注册系统账号。
- 激活码验证与激活。
- 非邮箱用户名登录。
- 代重置长辈密码。
- 账号删除接口。

- `mobile/lib/screens/login_screen.dart`
- `mobile/lib/screens/settings_screen.dart`
- `backend/app/routers/auth.py`

## 3. 当前不够完善的地方

### 3.1 真实上线前的安全与信任短板

健康类产品的第一要求不是功能多，而是用户信任。当前风险报告中已经指出几类需要优先处理的问题：

- Firebase Token 校验必须固定项目 ID，不能接受非本项目 token。
- 生产 JWT 默认密钥必须禁止启动。
- RevenueCat webhook 需要幂等和乱序保护。
- 激活码验证和激活需要限流、失败次数控制和审计。
- CORS 生产配置需要白名单。

产品影响：

这些问题一旦出现在真实用户环境，不只是技术事故，而是账号、家庭权限、付费权益和医疗隐私的信任事故。一人公司尤其承受不起这类事故。

### 3.2 弱网与失败反馈仍需产品化

当前已有离线同步队列，但风险报告指出移动端核心 HTTP 请求缺少统一超时，同步队列对永久性 4xx 错误也可能持续重试。

产品影响：

长辈用户遇到“转圈、没反应、打卡不知道成没成功”时，往往不会排查，只会放弃使用。健康执行类 App 的留存很大程度取决于失败时是否能继续完成动作。

建议把弱网体验作为产品功能处理：

- 打卡后先本地确认，再后台同步。
- 首页明确显示“已保存，稍后同步”。
- 同步失败用非技术文案提示。
- 登录、绑定、支付、报告生成必须有明确重试按钮。
- 401/403/404/422 与网络错误分开处理。

### 3.3 医疗数据生命周期不够一致

用药计划删除已采用 `is_active=False` 软删除，这是正确方向。但体征删除、用药日志取消、账号删除仍存在物理删除或历史事实丢失风险。

产品影响：

对慢病管理来说，历史连续性本身就是价值。用户误删、家属纠错、复诊追溯、付费报告都会依赖历史数据可信度。

建议：

- 血压/血糖删除改成软删除或“隐藏记录”。
- 用药打卡取消保留原始打卡事件，追加取消状态。
- 账号注销与医疗档案删除拆开，至少保留脱敏审计策略。
- 前端文案区分“停用计划”“删除显示”“注销账号”。

### 3.4 首页承载能力接近上限

`TodayScreen` 当前承载用药、体征、家属视角、报告、PRO、亲情语音、家庭入口等大量能力。对普通 App 这不一定是问题，但对银发用户是风险。

产品影响：

如果首页继续加入更多模块，长辈端会从“今天要做什么”变成“健康控制台”，破坏最核心的极简定位。

建议首页继续保持三层优先级：

1. 今天必须完成：服药、血压/血糖记录。
2. 今天需要关注：异常、漏服、家属提醒。
3. 低频入口：家庭、报告、设置、PRO。

复杂配置尽量放到家属端，长辈端只做执行、确认和少量查看。

### 3.5 运营和指标体系不足

从产品角度，一人公司不能只靠感觉判断是否有效。当前代码功能多，但缺少明确的产品指标闭环描述。

建议最少埋点或统计以下指标：

- 注册到首次创建用药计划转化率。
- 添加药物流程完成率。
- 今日用药打卡完成率。
- 血压/血糖 7 日记录频次。
- 家庭绑定成功率。
- 代注册激活完成率。
- 家属绑定后 7 日查看次数。
- 异常/漏服提醒打开率。
- 周报打开率。
- PDF 导出次数。
- 付费墙曝光、购买、恢复购买、失败率。

一人公司不需要复杂 BI，但需要每周能看懂：用户是否真的在用，家属是否真的在看，PRO 是否真的有价值。

## 5. 一人公司最应该补什么

一人公司最大的约束不是想法少，而是维护能力有限。因此优先级应遵循：

> 少加功能，多减少事故；少做大平台，多打通高频闭环；少做复杂 AI，多做家属愿意付费的确定价值。

### P0：上线前必须补

这些是影响安全、付费、数据可信和基础可用性的事项。

1. ~~修复 Firebase 项目 ID 校验与生产 JWT 默认密钥问题。~~ 已确认后端固定 `FIREBASE_PROJECT_ID` 校验，并在生产环境拒绝默认/过短 `SECRET_KEY` 启动。
2. ~~RevenueCat webhook 增加幂等和乱序保护。~~ 已按 `revenuecat_event_id` 去重，新增唯一索引，并避免旧过期时间覆盖新权益。
3. ~~激活码验证/激活增加 Redis 限流、失败次数和审计。~~ 已按 IP 与激活码维度增加失败计数、TTL 和 429 拦截。
4. ~~移动端 `ApiService` 统一请求超时和错误分类。~~ 已增加 20 秒统一超时、`ApiException` 结构化错误和同步队列 4xx 分类处理。
5. ~~付费权益矩阵落地：免费、本人 PRO、家庭共享 PRO 各能用什么。~~ 已将临床摘要接口切到家庭共享 PRO 口径，并明确权益矩阵见下表。
6. ~~体征删除和用药日志取消至少先建立产品规则，避免继续扩大物理删除路径。~~ 已新增体征软删除字段、用药取消审计字段，查询/报告/快照过滤已删除或已取消记录。
7. ~~首页打卡、体征录入、绑定、支付后刷新四条关键路径做一次端到端验证。~~ 已完成代码级验证：后端 `py_compile` 通过，`dart format` 通过，`flutter analyze` 无 error；真实设备全链路验证仍应在发版前按检查清单执行。

P0 权益矩阵：

| 能力 | 免费用户 | 本人 PRO | 家庭共享 PRO |
| --- | --- | --- | --- |
| 今日用药、打卡、基础体征记录 | 可用 | 可用 | 可用 |
| 家庭绑定与基础查看 | 可用 | 可用 | 可用 |
| 异常/漏服基础提醒 | 可用 | 可用 | 可用 |
| 周报、临床摘要、PDF 报告 | 不可用或受限 | 可用 | 可用 |
| 亲情语音提醒 | 不可用或受限 | 可用 | 可用 |
| PRO 家庭共享名额管理 | 不可用 | 可用 | 不可再次共享 |

P0 发版前真实设备检查清单：

- 首页打卡：在线打卡、离线打卡、恢复网络后同步、取消打卡后报告不计入已服药。
- 体征录入：在线/离线新增血压血糖、删除后首页/历史/周报/PDF 不再显示。
- 家庭绑定：邀请码、二维码、审核通过、解绑后权限消失。
- 支付刷新：购买成功、恢复购买、RevenueCat webhook 重复投递、旧事件乱序投递。

### P1：首批真实用户试点必须补

这些会影响留存、客服压力和家庭协同成功率。

1. 首页弱网状态：本地已保存、同步中、同步失败、点击重试。
2. 家庭绑定引导：清楚展示“谁邀请谁、谁能看到什么、如何取消”。
3. 添加药物流程继续简化：常用时间标签、饭前饭后标签、默认疗程。
4. 血压/血糖趋势：先做 7 天/30 天简单折线或列表摘要。
5. 异常值提示：使用“建议复测/必要时就医”，避免诊断式表达。
6. PDF 报告模板优化：医生优先看最近 30 天趋势、异常次数、用药依从率。
7. 最小运营看板：至少能每周导出核心指标。
8. 客服兜底文案：登录失败、绑定失败、支付失败、报告失败都有明确下一步。

### P2：验证付费后再做

这些有价值，但不应早于核心留存和付费验证。

1. 漏服原因记录。
2. 药盒/药品图片辅助识别。
3. 更多体征指标，例如体重、血氧。
4. AI 健康洞察。
5. 微信小程序或 Web 家属端。
6. 短信/电话提醒。
7. 多家庭/机构照护。
8. 医生端或诊所协作。

## 6. 不建议现在新增的功能

以下功能看起来有吸引力，但对一人公司当前阶段不划算：

- 医生端工作台：销售、合规、支持成本高，早期可先用 PDF 解决医生沟通。
- AI 诊断或疾病建议：医疗风险高，且数据量未必足够。
- 复杂处方规则引擎：会显著增加配置复杂度，破坏银发友好。
- 大而全健康档案：容易把产品从执行助手变成记录系统。
- 社区、内容、课程：偏离家庭慢病执行闭环。
- 免费版记录数量限制：会伤害真正需要多药共管的重度用户。

## 7. 建议的 30/60/90 天路线图

### 30 天：把产品变成可放心试点

目标：让 10-30 个真实家庭使用时，不因安全、支付、弱网和绑定问题频繁中断。

重点任务：

- 修复 P0 安全与付费风险。
- 统一移动端请求超时和错误文案。
- 梳理 PRO 权益矩阵并同步前后端。
- 对今日用药、体征记录、家庭绑定、支付恢复做冒烟测试清单。
- 建立最小指标记录方式。

成功标准：

- 首次创建用药计划流程可稳定完成。
- 今日打卡弱网下不会让用户丢失记录。
- 家庭绑定失败时用户知道下一步。
- 支付成功后权益能稳定刷新。

### 60 天：提升留存和家庭协同

目标：让长辈愿意每天打开，家属愿意每周查看。

重点任务：

- 优化首页信息层级，减少低频入口干扰。
- 添加 7 天/30 天血压血糖趋势。
- 优化家庭绑定确认页和解绑说明。
- 周报加入更强的家属可读摘要。
- PDF 报告模板面向复诊场景优化。
- 建立用户反馈入口和客服处理 SOP。

成功标准：

- 7 日留存有可观测数据。
- 家庭绑定成功率提升。
- 周报打开率和 PDF 导出次数能支撑 PRO 叙事。

### 90 天：验证商业化

目标：判断 PRO 是否值得继续投入，以及付费点是否需要调整。

重点任务：

- 将付费墙文案集中到“家人安心”和“复诊省心”。
- 对亲情语音、周报、PDF 三个权益做转化对比。
- 增加支付失败和恢复购买客服兜底。
- 小范围测试家庭共享名额和价格策略。
- 只在数据证明有效后，再考虑 AI 洞察或 Web 家属端。

成功标准：

- 付费墙曝光到购买有稳定漏斗数据。
- 至少一个 PRO 权益被真实用户高频使用。
- 客服问题集中在可修复流程，而不是产品方向不成立。

## 8. 最小可发布版本定义

如果以一人公司资源判断，一个务实的“可发布版本”不应追求功能更多，而应满足以下条件：

### 必须保留

- 登录/注册/激活码。
- 今日用药计划、打卡、取消打卡。
- 血压/血糖记录。
- 家庭绑定、审核、查看。
- 异常/漏服提醒。
- 周报或 PDF 至少保留一个强 PRO 权益。
- 设置、隐私、注销、基础法律页面。

### 必须稳定

- 首页加载。
- 打卡和体征保存。
- 弱网恢复同步。
- 家庭绑定。
- 支付成功和恢复购买。
- 账号找回或家属代重置。

### 可以延后

- AI 洞察。
- 多指标扩展。
- 医生端。
- Web 端。
- 复杂图表。
- 多机构照护。

## 9. 产品定位建议

当前最准确的定位仍然是：

> 面向长辈和慢病家庭的极简健康执行助手。

建议对外表达不要说“健康管理平台”，这会显得范围过大，也会带来用户对医疗能力的误解。更适合的表达是：

- 给长辈：每天只看今天该做什么。
- 给子女：不用天天追问，也能知道爸妈有没有按时吃药、体征是否异常。
- 给复诊：把最近一个月的用药和血压血糖整理成医生看得懂的报告。

## 10. 最终建议

Zellia 当前功能已经相当丰富。对一人公司来说，下一阶段最重要的不是“再做几个新功能”，而是把已有闭环做得足够可靠、足够清楚、足够容易付费。

建议优先顺序：

1. 先修安全、支付、弱网、数据删除和权限边界。
2. 再打磨首页、添加药物、体征录入、家庭绑定四条核心路径。
3. 然后用周报/PDF/亲情语音验证 PRO 转化。
4. 最后再考虑 AI、Web、医生端和更多健康指标。

当前产品可以进入小规模真实家庭试点；P0 代码加固已完成，但大范围商业化推广前仍应完成真实设备全链路验收和生产环境配置复核。
