# Zellia（岁月安）项目分析总结

> 生成日期：2026-05-28  
> 范围：根目录 `.cursorrules`、现有产品文档、后端 FastAPI 代码、Flutter 移动端代码、部署配置。

## 一、项目结论

Zellia 当前已经不是早期 Demo，而是一个 MVP+ 阶段的家庭协同健康管理产品。核心闭环包括：账号登录与激活、用药计划与今日打卡、血压/血糖记录、家庭绑定与守护、异常提醒、健康周报、临床 PDF 报告、PRO 订阅和家庭共享。

从技术上看，项目架构已经具备生产化雏形：后端有 FastAPI 分层、Alembic 迁移、Redis、Firebase、APScheduler、R2/S3、RevenueCat webhook；移动端有 Flutter、Firebase Auth/FCM、本地 SQLite、离线同步、桌面挂件和本地通知能力。

从产品上看，最清晰的差异化不是“健康数据记录”，而是“长辈可执行 + 家属可协同 + 医患沟通可导出”。这条路径符合 `.cursorrules` 中的定位：面向 60 岁以上用户的极简健康执行助手。

## 二、技术向分析

### 2.1 系统架构

项目采用移动端 + API 后端 + 外部服务的架构：

- 移动端：`mobile/`，Flutter 应用，入口在 `mobile/lib/main.dart`。
- 后端：`backend/`，FastAPI 应用，入口在 `backend/app/main.py`。
- 数据库：SQLAlchemy ORM，主目标为 PostgreSQL，Alembic 管理迁移。
- 缓存/临时状态：Redis，用于动态二维码 token、调度锁、冷却等。
- 认证：Firebase Auth 为主，legacy JWT 兜底，用于亲情激活等场景。
- 推送：Firebase Cloud Messaging。
- 支付：RevenueCat，后端有 webhook 与订阅事件记录。
- 文件/快照：R2/S3 相关服务用于周报冻结、语音或快照类对象。
- 部署：`docker-compose.yml` 中包含后端与 Redis，后端通过 `/health` 暴露 DB/Redis 健康状态。

整体分层比较清晰：路由在 `backend/app/routers`，业务服务在 `backend/app/services`，DTO 在 `backend/app/schemas`，移动端 API 聚合在 `mobile/lib/services/api_service.dart`。

### 2.2 核心数据模型

当前模型已经从 `.cursorrules` 的核心 5 表扩展为完整业务域：

- 用户：`User` 支持账号、昵称、邮箱、头像、系统代注册、激活码、会员状态、家庭语音。
- 用药：`MedicationPlan` + `MedicationLog` 支撑计划、时段展开、打卡记录、漏服提醒和软删除。
- 体征：`BloodPressureRecord`、`BloodSugarRecord` 支撑血压、血糖、幂等键和本地创建时间。
- 家庭：`FamilyLink`、`FamilyLinkActionLog` 支撑绑定、审核、别名、授权、操作留痕。
- 设备与推送：`DeviceToken`。
- 订阅与共享：`SubscriptionEvent`、`ProShare`。

数据模型方向合理，已经围绕“长期执行”和“家庭协同”形成主干。不过 `times_a_day` 仍是逗号分隔字符串，这对 MVP 简单有效，但长期会限制复杂服药规则，例如隔日、饭前饭后、每周特定日、疗程暂停等。

### 2.3 后端实现质量

后端有几个明显的成熟化改造痕迹：

- 启动生命周期中执行 Alembic upgrade，避免继续依赖运行时 DDL。
- APScheduler 定时任务使用 Redis 分布式锁，避免多 worker 重复执行。
- Redis/S3 客户端有复用封装，减少连接风暴。
- 今日用药接口已经做了批量查询当天日志，避免每个时段单独查库。
- 体征写入支持幂等键，适合移动端离线重试。
- 慢请求 profiling 有开关，适合线上问题排查。

主要技术风险：

- `SECRET_KEY`、CORS、Firebase 凭证、Redis URL 等生产配置必须强校验，避免默认值或宽松配置进入生产。
- `allow_origins=["*"]` 与 `allow_credentials=True` 不适合生产环境，应改为显式域名白名单。
- 核心授权逻辑分布在多个路由中的 `_resolve_target_user_id` / `_resolve_manage_target_user_id`，长期建议收敛成统一依赖或权限服务，降低遗漏风险。
- 家庭关系、用药日志、设备 token 的复合索引应继续补齐，尤其是高频首页和推送任务路径。
- Firebase Auth 是主要会话来源，服务不可用时会影响登录态验证；legacy JWT 只是部分兜底，不等于完整灾备。

### 2.4 移动端实现质量

Flutter 端已经覆盖多个复杂能力：

- 登录态恢复：Firebase currentUser 或 legacy JWT。
- 统一 API 客户端：自动拼接 Bearer token，401 触发登出。
- 今日页聚合：用药、体征、家庭成员数据、多用户切换。
- 本地能力：SQLite、本地通知、HomeWidget、语音提醒、离线同步。
- 商业化：RevenueCat 初始化、登录、登出、PRO 状态展示。
- 国际化：已有 `app_zh.arb`、`app_en.arb` 和生成文件。

主要移动端风险：

- `TodayScreen` 承载职责过重，已经同时处理用药、体征、家属视角、PDF、PRO、提醒冷却等逻辑，后续维护成本会上升。
- `ApiService` 同时包含底层 HTTP、DTO、业务 API、会话逻辑，文件会持续膨胀。
- 错误处理大多依赖调用方捕获，网络超时、503、弱网重试和用户提示仍有一致性空间。
- 文案和本地化需要持续收敛，避免部分页面硬编码中文。

### 2.5 工程与部署

项目具备基本部署能力：

- 后端 Dockerfile 与 `docker-compose.yml`。
- Redis 服务、健康检查、日志滚动。
- 后端通过 `.env` 注入配置。
- Firebase service account 通过 volume 挂载到容器路径。

需要注意：

- `docker-compose.yml` 中存在宿主机绝对路径挂载，换机器部署时容易失败，建议补充 `.env.example` 与部署说明。
- 生产环境需要明确 `DATABASE_URL`、`REDIS_URL`、`SECRET_KEY`、Firebase、RevenueCat、SMTP、R2/S3 等配置。
- `/health` 已覆盖 DB/Redis，但还可以增加外部服务配置缺失的诊断信息，避免静默跳过推送或周报。

## 三、产品向分析

### 3.1 产品定位

Zellia 的最佳定位是“家庭协同健康执行工具”，而不是泛健康记录 App。它面向的核心问题是：长辈每天需要做的事情很多，但复杂 App 会降低坚持率；家属想协助，但缺少可信的执行状态。

当前功能已经支持这个定位：

- 长辈端：按时吃药、记录血压血糖、查看今日状态。
- 家属端：绑定、审核、查看家人状态、接收提醒、周报、代注册和代重置密码。
- 医疗沟通：临床摘要 PDF 能把家庭记录转化为复诊材料。

### 3.2 用户角色

项目实际包含三类用户：

- 长辈或慢病管理者：核心使用者，要求交互极简。
- 家属/守护者：高频协同者，也是更可能付费的人。
- 医生或专业场景接收者：不一定登录产品，但会消费 PDF 报告或摘要结果。

产品设计应继续围绕这三类角色分层。长辈端不应承载复杂配置；配置、付费、报告、分享更适合家属端承担。

### 3.3 核心价值闭环

当前闭环可以概括为：

1. 建立账号和家庭关系：注册、代注册、激活码、邀请码、二维码。
2. 设置健康执行任务：用药计划、提醒、语音。
3. 每日执行：今日待办、打卡、体征录入。
4. 异常协同：漏服/异常体征推送给家属。
5. 周期复盘：每周健康周报、临床摘要 PDF。
6. 商业化：PRO、家庭共享、付费能力围绕家属安心和医患沟通展开。

这条闭环是健康类产品里比较完整的一条 MVP+ 路线，下一阶段重点不应盲目加功能，而应提高每个环节的稳定性和完成率。

### 3.4 差异化优势

- 银发友好：大字号、大按钮、少输入、低层级。
- 家庭协同：不仅记录本人数据，还解决家属授权、查看和协助。
- 账号兜底：代注册、激活码和代重置密码适合真实家庭场景。
- 医疗输出：PDF 临床摘要能连接家庭记录与复诊沟通。
- 情感提醒：亲情语音提醒比普通通知更有情绪价值。
- 商业化路径明确：基础执行免费，增值服务向子女和家庭收费。

### 3.5 产品风险

- 功能面已经较宽，长辈端首页需要避免变成“全功能入口集合”。
- 家属视角、只读视角、管理权限必须表达清楚，否则容易造成误操作。
- 健康异常提醒涉及信任边界，文案应避免医疗诊断口吻，更适合提示“建议关注/复测/必要时就医”。
- 订阅付费不要限制基础用药和基础记录，否则会伤害主目标用户留存。
- 亲情绑定、代注册、代重置密码是高信任链路，需要持续补操作留痕和反滥用策略。

## 四、建议优先级

### P0：生产安全与稳定

- 收紧 CORS 白名单。
- 强制生产环境配置 `SECRET_KEY`，禁止默认密钥启动生产实例。
- 补充 `.env.example` 和 Docker 部署说明，尤其是 Firebase 凭证路径。
- 为家庭授权、代重置、二维码绑定、订阅 webhook 增加关键测试。
- 补齐高频查询复合索引。

### P1：移动端可维护性

- 拆分 `TodayScreen`：用药区、体征区、家庭成员健康卡、报告导出、提醒冷却各自组件化。
- 拆分 `ApiService`：底层 HTTP client、DTO、业务 API 可以逐步分文件。
- 建立统一错误展示策略，覆盖网络超时、503、离线、鉴权失效。
- 清理硬编码文案，补全中英文 l10n。

### P2：产品增长与商业化

- 保持基础用药、基础体征记录免费。
- PRO 优先强化“子女安心”与“医患沟通”：实时提醒、周报、PDF、家庭共享、亲情语音。
- 增加 7/30 天趋势图和依从率趋势，提升家属端复盘价值。
- 增加医生友好报告模板或分享链接，但避免让长辈操作复杂化。

## 五、总体判断

Zellia 当前的产品方向成立，技术实现也已经从简单 CRUD 走向家庭协同、推送、离线、订阅、报表和生产部署。下一阶段最重要的是收敛复杂度：后端收敛权限与配置安全，移动端拆分重型页面与 API 文件，产品端守住银发友好原则，把商业化集中在家属安心和复诊沟通上。
